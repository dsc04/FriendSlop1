# Онлайн-мультиплеер: кооп на 2–8 игроков

Пошаговый план, как сделать нашу игру **кооперативной онлайн-игрой на 2–8
игроков**, и — самое важное — **как писать персонажа «сетевым» с самого начала**,
чтобы потом ничего не переделывать.

> TL;DR
> - **Netcode for GameObjects (NGO)** — сетевой код.
> - **Multiplayer Services / Sessions API** (внутри — Relay + Lobby +
>   Authentication) — подключение по интернету по коду, без проброса портов.
> - Модель: **хост-клиент** — один игрок = хост (сервер+клиент), остальные
>   подключаются к нему. Отдельный сервер (dedicated) нам НЕ нужен.
> - Начинаем с «владелец управляет своим персонажем», общий мир живёт на хосте.

---

## 1. Что мы выбираем и почему

| Слой | Выбор | Почему |
| --- | --- | --- |
| Сетевой код | **Netcode for GameObjects (NGO)** | Официальный, первый-party, заточен под кооп и наш обычный GameObject-проект. У нас уже стоит пакет **Multiplayer Center**, который его и ставит. |
| Транспорт | **Unity Transport (UTP)** | Ставится вместе с NGO. |
| Связь по интернету | **Relay** (+ **Lobby**) через **Sessions API** | Игроки соединяются по коду, без белого IP и проброса портов. |
| Вход | **Authentication** (анонимный) | Нужен для Relay/Lobby. Игрок ничего не регистрирует. |
| Топология | **Хост-клиент** (listen server) | Для кооп 2–8 — самый простой и бесплатный вариант. |

**Чего НЕ берём (пока):** dedicated server, Netcode for Entities (DOTS),
сторонние Photon/Mirror/FishNet. Для нашего кооп это лишняя сложность.

**Авторитет (кто «главный» над состоянием):**
- **Движение своего персонажа** — у владельца (owner authority). Проще и
  отзывчивее.
- **Общий мир** (враги, двери, предметы, урон, счёт) — на **хосте/сервере**.
- Кооп = играем с друзьями, читы не критичны, поэтому можно не делать всё
  строго server-authoritative. Важную логику (урон, прогресс) всё равно держим
  на хосте.

---

## 2. Как писать персонаж СЕТЕВЫМ (читать тому, кто пишет персонажа!)

Даже если сети ещё нет — пиши персонажа сразу по этим правилам. Тогда включить
мультиплеер будет легко.

**Главные правила:**
1. Скрипт персонажа наследуем от **`NetworkBehaviour`** (а не `MonoBehaviour`),
   на корне префаба — компонент **`NetworkObject`**.
2. **Ввод читает только владелец.** В начале `Update()`:
   `if (!IsOwner) return;` — чужими персонажами локально не управляем.
3. **Камера и локальный ввод — только для своего игрока** (`IsOwner`). У каждого
   на экране своя камера, но персонажей на сцене несколько.
4. **Отделяй «намерение» от «симуляции».** Ввод (нажатия) → в одном месте,
   применение к физике/движению → в другом. Так проще потом перенести проверку
   на сервер.
5. **Не полагайся на `Camera.main` и «единственного игрока».** Игроков много.
6. **Действия, влияющие на общий мир** (выстрел, открыть дверь, нанести урон) —
   отправляй на сервер через **RPC**, а не меняй мир напрямую у себя.
7. **Общее состояние** (здоровье, счёт) храни в **`NetworkVariable<T>>`**, а не в
   обычных полях — тогда оно само синхронизируется всем.
8. **Персонажа не ставим в сцену вручную** — его спавнит NetworkManager как
   Player Prefab на каждого подключившегося.

**Мини-пример (шаблон, детали — в доке NGO):**

```csharp
using Unity.Netcode;
using UnityEngine;

public class PlayerController : NetworkBehaviour
{
    // Общее состояние: пишет только сервер, читают все
    readonly NetworkVariable<int> _health = new(
        100, NetworkVariableReadPermission.Everyone,
             NetworkVariableWritePermission.Server);

    public override void OnNetworkSpawn()
    {
        // Камеру/ввод включаем только своему игроку
        if (IsOwner) EnableLocalCameraAndInput();
    }

    void Update()
    {
        if (!IsOwner) return;          // чужих не двигаем
        // читаем ввод и двигаем СВОЕГО персонажа
    }

    // Действие, меняющее общий мир — выполняем на сервере
    [Rpc(SendTo.Server)]
    void FireRpc(Vector3 direction)
    {
        // тут стреляем/наносим урон на хосте, результат разойдётся клиентам
    }
}
```

> `[Rpc(SendTo.Server)]` — современная форма (NGO 2.x). Классические
> `[ServerRpc]` / `[ClientRpc]` тоже работают.

**Чек-лист «сетевого персонажа»:**
- [ ] На корне префаба есть `NetworkObject`.
- [ ] Скрипт — `NetworkBehaviour`.
- [ ] Ввод/камера под `IsOwner`.
- [ ] Движение — через `NetworkTransform` (режим авторитета **Owner**).
- [ ] Синхронное состояние — в `NetworkVariable`, а не в обычных полях.
- [ ] Воздействие на мир — через `[Rpc(SendTo.Server)]`.

## 3. Установка пакетов

**Проще всего — через Multiplayer Center** (пакет уже стоит):
1. `Window ▸ Multiplayer ▸ Multiplayer Center`.
2. Тип игры — **Cooperative** (кооп).
3. Нажать установку рекомендованных пакетов — он поставит **NGO**, транспорт и
   мультиплеер-сервисы.

**Или вручную** (`Window ▸ Package Manager`): `Netcode for GameObjects`,
`Multiplayer Services`, `Authentication`.

---

## 4. Подключить Unity Gaming Services (UGS)

Relay/Lobby работают через облачные сервисы Unity (есть бесплатный лимит,
для игры с друзьями его хватает).

1. Создать/привязать проект на **dashboard.unity.com**.
2. Включить сервисы: **Authentication**, **Relay**, **Lobby**.
3. В Unity: `Edit ▸ Project Settings ▸ Services` — связать проект с UGS.

---

## 5. Базовая сетевая сцена

1. Пустой объект **`NetworkManager`** + компонент **Unity Transport**.
2. Поле **Player Prefab** → наш префаб игрока (из раздела 2).
3. Ограничить число подключений: **макс. 8**.
4. Смену сцен делаем через **NetworkManager (NetworkSceneManager)** —
   загрузку сцены инициирует хост, она синхронно грузится у всех.

---

## 6. Поток подключения (Host / Join по коду)

Через **Sessions API** (обёртка над Relay + Lobby):

```
Хост:    вход (анонимно) → создать сессию → получить КОД → StartHost
Клиент:  вход (анонимно) → ввести КОД → присоединиться к сессии → StartClient
```

- Хост создаёт сессию — под капотом выделяется **Relay** и создаётся **Lobby**,
  наружу отдаётся короткий **код подключения**.
- Клиент вводит код → подключается через Relay к хосту.
- Никаких белых IP и проброса портов не нужно.

Мини-UI для старта: поле «Ввести код», кнопки **Host** и **Join**.

---

## 7. Общий мир (враги, предметы, двери)

- Объекты мира — это **`NetworkObject`**, их **спавнит сервер/хост**
  (`GetComponent<NetworkObject>().Spawn()`), клиенты получают копии.
- Их логика (ИИ, урон, состояние) считается **на хосте**.
- Клиент просит что-то сделать через **`[Rpc(SendTo.Server)]`**, хост проверяет
  и меняет состояние; изменения расходятся автоматически.
- Синхронизируем **только нужное** (позиции, здоровье, статусы) — не пихаем в
  сеть всё подряд, иначе трафик и лаги.

---

## 8. Жизненный цикл игроков

- Подключение/выход ловим на `NetworkManager` (`OnClientConnected` /
  `OnClientDisconnected`).
- Player Prefab спавнится автоматически на каждого; респавн — на хосте.
- **Если вышел хост** — сессия завершается. Полноценную «миграцию хоста» для
  кооп на старте НЕ делаем (это сложно) — просто корректно завершаем матч и
  возвращаемся в меню.

---

## 9. Тестирование

- **Multiplayer Play Mode (MPPM)** — `Window ▸ Multiplayer ▸ Multiplayer Play
  Mode`. Запускает до **4 игроков** прямо в редакторе (1 основной + 3
  виртуальных) — идеально для быстрой проверки.
- **ParrelSync** (community) — делает клон проекта, если нужно больше окон.
- Порядок: сначала локально (MPPM), потом **сборкой по интернету** через Relay
  (проверить с друга из другой сети).

## 10. Пошаговый план по этапам

Двигаемся маленькими проверяемыми шагами — не пытаемся сделать всё сразу.

- **Этап 0 — Подготовка.** Поставить пакеты (разд. 3), привязать UGS (разд. 4).
  _Готово, когда:_ проект собирается, сервисы включены.
- **Этап 1 — Два игрока в одной сцене.** NetworkManager + Player Prefab с
  `NetworkTransform`. Запустить в MPPM, увидеть, как двигаются оба.
  _Готово, когда:_ в MPPM два персонажа видят движение друг друга (пока без
  интернета, локально).
- **Этап 2 — Подключение по коду.** Прикрутить Sessions API: Host/Join по коду
  (разд. 6) + мини-UI.
  _Готово, когда:_ друг из другой сети заходит по коду.
- **Этап 3 — Сетевой персонаж.** Довести персонажа по разделу 2: ввод владельца,
  анимации, действия через RPC, здоровье в `NetworkVariable`.
  _Готово, когда:_ действия персонажа корректно видны всем.
- **Этап 4 — Общий мир.** Враги/предметы спавнятся и живут на хосте (разд. 7).
  _Готово, когда:_ все видят одинаковый мир и взаимодействие.
- **Этап 5 — Лобби и жизненный цикл.** UI лобби, join/leave, респавн (разд. 8).
  _Готово, когда:_ можно собрать сессию 2–8 и доиграть до конца.
- **Этап 6 — Полировка и релиз.** Интерполяция движения, обработка потерь связи,
  сборка, тест по интернету на 2–8.
  _Готово, когда:_ играется стабильно на реальном пинге.

## 11. Подводные камни

- **Не пиши персонажа как чисто одиночного** — потом больно переделывать
  (см. разд. 2, это главное).
- **Не начинай с dedicated server / DOTS netcode** — для кооп это оверкилл.
- **Не синхронизируй всё подряд** — только то, что реально нужно всем.
- **Хост-авторитет.** Урон/прогресс/важную логику считаем на хосте, иначе рассинхрон.
- **Меняй сцены только через NetworkManager**, а не обычным `SceneManager`.

## 12. ✅ Уже сделано в проекте — как запустить (июль 2026)

Этапы 0–2 из плана выше реализованы: пакеты прописаны в проекте, есть
меню Host/Join по коду. Код лежит в `Assets/_Project/Scripts/Network/`.

**Один раз (только владелец проекта):** проверить привязку к Unity Cloud —
`Edit ▸ Project Settings ▸ Services`: должен быть выбран проект. Если пусто —
войти в Unity-аккаунт и привязать (Link).

**Собрать сцену (один раз):** `Tools ▸ FriendSlop ▸ Set Up Multiplayer Scene` —
создаст префаб игрока, NetworkManager и меню подключения. Если сцены ещё нет,
сначала `Tools ▸ FriendSlop ▸ Build Foundation Scene`.

**Играть вместе:**
1. Хост: Play → кнопка **HOST a room** → появится короткий **код комнаты**.
2. Друг: `git pull` → открыть проект → Play → ввести код → **JOIN**.
3. Никаких портов и белых IP не нужно — соединение идёт через Unity Relay
   (бесплатно).

**Потестить одному:** `Window ▸ Multiplayer ▸ Multiplayer Play Mode` → включить
Player 2 → Play. Два окна = два игрока на одном компьютере.

**Максимум игроков:** поле *Max Players* на объекте `ConnectionMenu` в сцене
(сейчас 6).

**Автору персонажа (FP-версия уже влита):** `PlayerController.cs` и
`PlayerInputHandler.cs` мультиплеер НЕ менял. Префаб игрока собирается
инструментом из тех же частей, что и `FoundationSceneBuilder` (PlayerBody,
камера-ребёнок, PlayerInputHandler на корне), плюс сетевые компоненты.
`NetworkPlayerSetup.cs` на префабе включает управление/ввод/камеру только
владельцу — если переименуешь классы или добавишь свои скрипты, обнови
помеченные места там. В сцене есть `MenuCamera` (вид до подключения),
**Esc** разблокирует курсор во время игры. Дальше по плану — раздел 2.
