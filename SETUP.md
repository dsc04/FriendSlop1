# Настройка проекта и работа с Git

Документация по подключению к проекту, структуре папок и правилам совместной
работы через Git. Проект: **Unity 6 (6000.5.2f1)**.

---

## 1. Что нужно установить

- **Unity 6000.5.2f1** — ставится через **Unity Hub**. Важно использовать ту же
  версию, что указана в `ProjectSettings/ProjectVersion.txt`.
- **Git** — https://git-scm.com
- **Git LFS** — хранит крупные бинарные файлы (арт, звук, модели):
  - macOS: `brew install git-lfs`
  - Windows: https://git-lfs.com (или `winget install GitHub.GitLFS`)

После установки Git LFS выполните **один раз** на своём компьютере:

```bash
git lfs install
```

---

## 2. Как склонировать проект

```bash
git clone <URL-репозитория>
cd <папка-проекта>
git lfs pull        # догрузит бинарные файлы из LFS (обычно автоматически)
```

Затем откройте папку проекта через **Unity Hub** нужной версией Unity.

---

## 3. Настройки Unity для контроля версий

Уже настроены в проекте — менять не нужно, но полезно знать, где это:

- **Asset Serialization → Force Text**
  `Edit ▸ Project Settings ▸ Editor ▸ Asset Serialization ▸ Mode`
  Делает ассеты текстовыми: читаемый diff и возможность слияния.
- **Version Control → Visible Meta Files**
  `Edit ▸ Project Settings ▸ Version Control ▸ Mode`
  Каждый ассет получает файл `.meta`.

> ⚠️ **Файлы `.meta` — часть проекта.** Всегда коммитьте `.meta` вместе с самим
> ассетом. Перемещайте и переименовывайте ассеты **только внутри Unity**
> (в окне Project), иначе ломаются ссылки между объектами.

## 4. Структура папок

Всё, что делает команда, складываем в `Assets/_Project/`:

```
Assets/
  _Project/
    Scenes/        — сцены
    Scripts/       — C#-скрипты
    Prefabs/       — префабы
    Art/
      Sprites/     — 2D-графика
      Models/      — 3D-модели
    Audio/         — звук и музыка
    Materials/     — материалы
    UI/            — интерфейс
  ThirdParty/      — импортированные пакеты и ассеты со стороны
```

Подчёркивание в `_Project` держит папку вверху списка. Сторонние ассеты —
только в `ThirdParty/`, чтобы не смешивать с нашим кодом.

Исходники графики (`.blend` и т.п.) — в папке `ArtSource/` в корне проекта
(вне `Assets/`, см. раздел 9).

---

## 5. Что НЕ попадает в Git

Папки `Library/`, `Temp/`, `Obj/`, `Build/`, `Logs/`, `UserSettings/`
генерируются автоматически и исключены в `.gitignore`. Коммитить их не нужно.

---

## 6. Git LFS

Крупные бинарники (`.png`, `.psd`, `.fbx`, `.wav`, `.ogg`, `.mp4`, шрифты и т.д.)
автоматически хранятся в Git LFS — правила заданы в `.gitattributes`. Ничего
дополнительно делать не нужно, коммитьте как обычно.

Проверить, что файл ушёл в LFS:

```bash
git lfs ls-files
```

---

## 7. Рабочий процесс (Git-флоу)

Используем простой **GitHub Flow** — его достаточно для небольшой команды.

**Правило №1:** ветка `main` всегда рабочая (проект открывается и запускается).

Порядок работы над задачей:

1. Обновить main: `git checkout main && git pull`
2. Создать ветку под задачу:
   `git checkout -b feature/имя-задачи` (или `fix/…` для багов)
3. Работать и коммитить небольшими понятными шагами:
   `git add -A && git commit -m "Короткое описание"`
4. Отправить ветку: `git push -u origin feature/имя-задачи`
5. Открыть **Pull Request** на GitHub, попросить коллегу быстро взглянуть.
6. После одобрения — влить в main (лучше **Squash and merge**) и удалить ветку.
7. Остальным — `git pull` в main, чтобы быть в актуальном состоянии.

```
main  ──●───────────────●─────────────▶   (всегда рабочая)
         \             /
          ●──●──●──●                        feature/…  (короткая ветка + PR)
```

### Важно для Unity (чтобы не ловить конфликты)

- **Сцены и префабы плохо мёржатся.** Договаривайтесь, кто сейчас правит
  конкретную сцену/префаб. По возможности выносите части в отдельные префабы.
- **Пуллите часто, коммитьте маленькими порциями.** Чем реже синхронизация —
  тем больнее конфликты.
- Не коммитьте, пока Unity импортирует ассеты — дождитесь окончания.
- Пишите короткие, но понятные сообщения коммитов (что сделано).

---

## 8. Полезные команды

| Действие              | Команда                                   |
| --------------------- | ----------------------------------------- |
| Статус                | `git status`                              |
| Обновить main         | `git checkout main && git pull`           |
| Новая ветка           | `git checkout -b feature/…`               |
| Закоммитить всё       | `git add -A && git commit -m "..."`       |
| Отправить ветку       | `git push -u origin <ветка>`              |
| Что ушло в LFS        | `git lfs ls-files`                        |

## 9. Для художников (модели в Blender)

Если ты занимаешься только моделями в Blender — тебе **не нужно открывать Unity
и учить командную строку**.

**Формат:** работаем в `.blend`, но в игру отдаём **FBX**.

Куда класть файлы:

- Исходник `.blend` → `ArtSource/Models/` (в корне проекта, Unity её не трогает).
- Экспортированный `.fbx` → `Assets/_Project/Art/Models/` — его использует игра.
- Текстуры → `Assets/_Project/Art/Sprites/` (или рядом с моделью).

> ⚠️ **Не клади `.blend` внутрь `Assets/`** — Unity попытается его импортировать
> и потребует установленный Blender у каждого в команде. В `Assets/` — только FBX.

**Проще всего работать через GitHub Desktop** — https://desktop.github.com

- Ставит Git и Git LFS сам, командная строка не нужна.
- Один раз: _Clone repository_ → выбрать `dsc04/FriendSlop1`.
- Дальше цикл: сделал модель → экспортировал FBX в папку проекта →
  в GitHub Desktop появятся изменения → пишешь описание →
  _Commit_ → _Push_ → _Create Pull Request_.

**Советы по экспорту (Blender → Unity):**

- Перед экспортом применяй трансформации: `Object ▸ Apply ▸ All Transforms`
  (`Ctrl+A`), иначе модель приедет повёрнутой или увеличенной.
- Договоритесь о масштабе: 1 юнит Unity = 1 метр.
- Backup-файлы Blender (`.blend1`, `.blend2`) в Git не попадают — они в `.gitignore`.
